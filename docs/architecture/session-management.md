# Session Management

This document describes the gateway layer and session lifecycle for the Narrative Network orchestrator.

---

## Gateway

The gateway is defined in `orchestrator/gateway.py`. It is a FastAPI application and the only internet-facing component of the system. All client traffic enters and exits through it.

### Endpoints

| Method | Path | Description |
|---|---|---|
| `POST` | `/enter` | Start a new session |
| `POST` | `/hop` | Advance the current session to a new node |
| `GET` | `/session/{id}` | Retrieve session state |
| `WS` | `/session/{id}/live` | Stream hop events over WebSocket |
| `GET` | `/healthz` | Health check |

### CORS

CORS middleware is configured via `SubnetConfig`. Allowed origins, headers, and methods are derived from subnet configuration rather than being hardcoded.

### Session Registry

Sessions are stored in an in-memory registry for the current deployment. Redis is planned as the backing store when multi-instance high availability is required.

## Request and Response Models

### EnterRequest

| Field | Type | Constraints |
|---|---|---|
| `query` | string | 1 to 2000 characters |
| `persona` | string | Optional |
| `max_hops` | integer | 1 to 20 |
| `entry_node_id` | string | Optional override; bypasses router resolution |

### EnterResponse

| Field | Description |
|---|---|
| `session_id` | Unique identifier for the new session |
| `entry_node_id` | The node the session entered |
| `entry_narrative` | Opening narrative text generated by the entry node |
| `path` | Ordered list of node IDs visited so far |
| `available_next_nodes` | Node IDs the client may hop to next |

### HopRequest

| Field | Description |
|---|---|
| `session_id` | Session to advance |
| `to_node_id` | Target node for this hop |

### HopResponse

| Field | Description |
|---|---|
| `session_id` | Session identifier |
| `from_node_id` | Node the session hopped from |
| `to_node_id` | Node the session hopped to |
| `hop_text` | Narrative text generated for this hop |
| `path` | Full path including this hop |
| `available_next_nodes` | Node IDs available for the next hop |
| `is_terminal` | True if the session has reached a terminal state |

### SessionResponse

| Field | Description |
|---|---|
| `session_id` | Session identifier |
| `path` | Ordered list of all visited node IDs |
| `narrative_so_far` | Accumulated narrative text across all hops |
| `current_node_id` | The node the session is currently at |
| `is_terminal` | True if the session has reached a terminal state |
| `hop_count` | Number of hops completed |

## OrchestratorSession

Session logic lives in `orchestrator/session.py`.

### Session States

```
CREATED --> ACTIVE --> TERMINAL
                  \--> ERROR
```

### Lifecycle

A session is driven by a sequence of `enter()` and `hop()` calls.

#### enter()

1. Embed the client query.
2. Resolve the entry node via the Router, or use the `entry_node_id` override if provided.
3. Fetch corpus chunks from the domain miner via `KnowledgeQuery`.
4. Generate the opening narrative via a `NarrativeHop` self-loop (the node narrates its own domain as the entry point).
5. Run a safety guard tick.

#### hop()

1. Validate that `to_node_id` is present in `available_next_nodes`.
2. Embed the accumulated narrative to capture semantic context.
3. Fetch corpus chunks from the target domain miner via `KnowledgeQuery`.
4. Generate the hop narrative via `NarrativeHop`.
5. Append the new node to the session path.
6. Run a safety guard tick; the guard may modify the hop text or reroute the session.
7. Log quality metrics to the graph store (`word_count / MAX_HOP_WORDS` logged via `graph_store.log_traversal`).
8. Check for terminal conditions: `hop_count >= max_hops` or no reachable next nodes.

### Continuity Invariant

The orchestrator guarantees that no session ever reaches a dead end as a result of graph mutation. `safety_guard.filter_candidates()` screens the set of reachable next nodes before they are returned to the client. `safety_guard.tick()` may modify hop text or silently reroute the session when graph mutations (such as pruning) would otherwise leave a session stranded.

## WebSocket Streaming

Clients that connect to `WS /session/{id}/live` receive hop events in real time rather than polling the REST endpoints.

### Client Messages

```json
{ "to_node_id": "<node_id>" }
```

### Server Events

The server emits a hop event for each completed hop, structured as a `HopResponse`. When the session reaches a terminal state, the server emits a terminal event and closes the connection.

The connection is also closed if the client disconnects.

---

## Safety Guard

The `PathSafetyGuard` is responsible for maintaining session integrity across all graph mutations.

### filter_candidates()

Called before `available_next_nodes` is returned to the client. Removes any candidate node that is unreachable, decayed below the visibility threshold, or otherwise unsafe for the current session context.

### tick()

Called after each hop is generated. May take the following actions:

- Modify the hop text to include a safety bridge — narrative language that prepares the reader for a domain transition caused by graph mutation.
- Reroute the session to an alternative node if the originally planned next node has become unavailable since the last hop.

These actions are transparent to the client: the session path and narrative remain coherent, and no error is surfaced for graph-mutation-driven reroutes.
